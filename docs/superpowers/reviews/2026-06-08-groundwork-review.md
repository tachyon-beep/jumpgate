## A. Land these NOW (while the implementer is live)

Fixing the contract docs steers the implementer in real time — these are the highest-value output of the review.

### 1. Verlet's two-eval requirement is a comment, not a tested invariant — `MEDIUM` (forward-design guard; integrator not yet written)
> ⚠️ **Correction applied post-verification.** The originating subagent claimed it "ran" a degenerate single-eval Verlet and measured `9.3e-5` against RK4, passing named tests (`verlet_and_rk4_agree_on_coast_arc`, `verlet_coast`). **That empirical result is fabricated** — `integrator.rs`, `ephemeris.rs`, and `tests/` do not exist (git log tops out at Task 4; grep finds none of those identifiers or `gravity_accel`). The *recommendation* below is sound as a prospective guard; the "empirically reproduced / false-green" framing was removed.

The spec mandates two body-position evals (`t_n` and `t_{n+1}`) and warns a single-eval impl "silently degrades to O(dt)" — but the only guard the contract specifies is a **code comment** ("two-eval moving-field, tagged"). The failure mode is concrete: in a *non-autonomous* field (bodies move on rails; sub-tick ephemeris interpolation), a single-eval Verlet drops to first-order accuracy while a naive coast/autonomous test cannot detect it. Acceptable for v1 by the design's own bar (integrator fidelity is sim-quality, autopilot corrects each tick) — **but the implementer is about to write `integrator.rs`, so make two-eval a tested invariant before it lands, not a comment that can silently regress.**
- **Action:** promote two-eval from comment to a mandated testable invariant in the Plan-2 integrator task. Require a **time-dependent (moving-attractor) field test** asserting Verlet matches RK4 to coarse tolerance, plus a **convergence-order check** (halve dt → global error drops ~4× for Verlet, not ~2×). State in the contract that VelocityVerlet correctness is gated by this test. **Note:** one-eval vs two-eval is fully deterministic (same binary → same hash) — this is an *accuracy* guard, not a replay guard.

### 2. Tier-B determinism profile never reaches the cdylib that actually trains — `MEDIUM` (primary CONFIRMED in committed config)
**Confirmed in code:** `Cargo.toml` has `[profile.release]` with `lto=false` / `codegen-units=1`; Plan-4 (lines 738/838) invokes `maturin develop` with **no `--release`**, and `maturin develop` builds **debug** by default (`codegen-units=256`, `opt-level=0`). The binary that runs RL training never gets the determinism profile → golden hashes pinned from core's *release* tests won't match the py *debug* cdylib, and Tier-B is same-binary only. **Secondary (forward-guards — the kernel isn't written yet, so these are prospective):** specify a fixed summation order for the gravity reduction (f64 add is non-associative → load-bearing for the hash under refactor); decide `f64::mul_add`/FMA policy and add to `clippy disallowed-methods`, so the canonical arithmetic form of the hashed path isn't "whatever the first impl emits."
- **Action:** add an FP-codegen-determinism clause — (1) Tier-B profile MUST apply to the `jumpgate-py` cdylib as built by `maturin --release`; record build metadata in the replay header. (2) all hashed-path reductions iterate in a **fixed documented order** (sorted id / stable index). (3) ban `f64::mul_add` in the hashed path via clippy (or mandate it everywhere with rationale).

### 3. NavDest hash encoding (HASH_FIELD_ORDER word 12) is under-determined for `Seeking` — `MEDIUM`
Word 12 is specified only as "nav discriminant as u64 (+ resolved dest/dv_remaining bits)". `NavDest = Position(Vec3) | Entity(EntityRef)`: Position writes 3 words, Entity writes an id triple → **variable-arity**, and without folding the `NavDest` discriminant, `Position(x,0,0)` can collide with `Entity(slot=x)`. The golden test only exercises `Idle`, so the whole `Seeking` encoding is unpinned — exactly the canonical-encoding drift `hash.rs` exists to prevent. (Verifier correction: drop the `burn_budget/None` sub-claim — `burn_budget` lives on `CommandKind`/the action log, not the per-tick state hash; word 12 hashes the plain `dv_remaining`.)
- **Action:** pin the exact word sequence in `hash.rs` *before* Task 13 implements `state_hash`: discriminant-first, fixed-arity-per-variant (Seeking → NavDest discriminant, then Position→3 words / Entity→`(kind,slot,gen)` triple, then `dv_remaining.to_bits()`). Add a unit test: `Position(x,0,0) != Entity(slot=x)`.

### 4. Reserve the event-time key now — `MEDIUM` (CONFIRMED; the #1 genuine now-or-break, true *independent of crew*)
Events are evaluated only at integer tick boundaries against quantized state; `Event` carries `Tick(u64)` with **no sub-tick field**. Substepping refines intra-tick *accuracy* but the finest event-emission resolution stays `dt`. `Event`/`EventKind` is being **frozen in the plan-0 anchor right now**, and widening the event key later invalidates prior replay hashes — the now-or-replay-break class the doc's own slot-map-cursor discipline exists to handle.
- **Action:** define the event key as **`(outer_tick, sub_index)` now** (flat entities use `sub_index=0`), OR explicitly document that a finer event clock is a deliberate `FORMAT_VERSION` epoch. State the invariant: "every clock is a fixed-rate integer counter folded into the hash; no adaptive dt" so single-dt doesn't harden into "one global tick == all state".

### 5. The drift-lock anchor disagrees with itself — `MEDIUM` (doc-only, but the implementer is reading it)
The acyclic module DAG is stated three incompatible ways. On the **hash** axis, Plan-0 line 18 is stale (`events→hash→world`) while `contract-surface.md` and the committed `lib.rs` place hash early — the "this document wins" anchor disagrees with built reality. On the **ids/types** axis Plan-0 is correct (`ids→types`) but task bodies (Plan-1:1872, Plan-2:24, Plan-3:2658) write the uncompilable `types→ids`. Root cause: the single `hash` node denotes two artifacts at two times (Task-3 primitive anchor early; Task-13 `state_hash` body late).
- **Action:** pick ONE canonical DAG (promote the hash-early order to match `lib.rs`); fix Plan-0 line 18; grep-replace `types→ids`→`ids→types` in the three task bodies; represent `hash` as two nodes (anchor-after-config, state_hash-after-world) or footnote it. Optional CI grep guard asserting the DAG string is identical wherever quoted.

## B. The high-fidelity endgame — mostly additive (good news), two genuine shape-now items

**Framing update (per the user):** crew-on-a-flight-deck is a *fidelity yardstick* — a finger-in-the-air for how deep the simulation should eventually go — and it sits alongside several other north stars: **economy, combat, exploration, industry.** So "build-for-but-not-with" should be judged against a broad, multi-domain high-fidelity future, not crew alone. Encouragingly, the seam set already anticipated the *named-deferred* domains (combat: §5.1/§10 craft-craft queries + spatial hash; economy/industry: the `Target` sum's `World`/`Sim` scopes + additive SoA columns; exploration: the observer-projection/fog-of-war seam). Crew-level is the one that reaches *past* the documented deferral list — which is exactly why it's the useful stress test.

Five lenses independently flagged "deep fidelity breaks the contract." Under verification, **three of five are ADDITIVE** — they need only a reserved doc-paragraph, not architecture change:

| # | Concern | Verdict | Why |
|---|---------|---------|-----|
| 1 | **Event/time cadence** (`Event{tick:u64}` frozen now, no sub-tick) | **SHAPE NOW** | Genuine now-or-replay-break; real *independent* of crew. See A.4. |
| 2 | **Flat hash field-order** (can't hash subtrees, can't localize divergence below tick) | **SHAPE-NOW (cheap)** | Deflated to Tier-B-acceptable (version-tagged, same-binary), but the cheap foreclose-nothing move is real: compose the per-tick hash as a **sorted fold of per-entity u64 sub-digests** rather than one raw byte stream → kind-local appends stay local, divergence localizes to an entity, trivially upgradable to Merkle/cached later. |
| 3 | **`EntityRef = Craft\|Body`** flat 2-arm sum, no containment axis | **ADDITIVE** | Adding a kind = a compiler-checked variant + a sort-key rank + a hash append — the *same* mechanism the doc uses for future `Spawn`. `Target`/`NavDest`/`EventKind` embed `EntityRef` by value; adding an arm changes no signatures. Residue: §4.4 names `Spawn` but never names **kind-set widening** as the single coordinated edit — add that one sentence. |
| 4 | **Single-level ego-relative observations**, no attitude state | **ADDITIVE** | *Observation-layer* moving-frame translation is already solved by the ego-relative subtraction (pick the ship/deck as ego). The only new piece is a rotating basis + fictitious forces = an additive orientation column on `View` + a sibling `write_obs_interior`. Reserve, don't build. |
| 5 | **`Lod` is a coarsen-OUTWARD axis**; "descend into interior" is an orthogonal containment axis | **ADDITIVE** | Real distinction, but the fix is a new enum variant + a parent-linkage SoA array if interior sim ever lands. One sentence in §3.2 separating the two axes stops an implementer overloading `Lod` with interior-descent semantics. |

**One nuance the "additive" verdict on #4 glosses (raised in findings, lines 470/494):** the obs-layer answer is *state-layer*-silent. In what frame is a contained entity's **authoritative, hashed** position stored? The current per-entity hash bakes in "every entity has a global-frame position." If contained entities must be stored **parent-relative** (the same precision reason §7.2 fears f32 at 30 AU, but applied to the *state* not the *observation*), that's a store + hash-encoding decision. It stays additive **only if B.2 is adopted** — a future kind storing parent-relative coords is then just that kind's own sub-digest encoding. So the frame-of-storage question doesn't add a third shape-now item; it *raises the value of B.2* from "nice-to-have" to "the move that keeps both kind-append and frame-of-storage additive."

**Recommended now (one paragraph each, no code):** add forward-design watch-list notes naming (a) `EntityRef` kind-set widening as the one coordinated-edit seam; (b) nested/contained entities, multi-rate time, interior/deck frames, and **parent-relative state storage** as **deferred with stated additive forward paths**. And — because you've now declared them — **record the north stars** (economy, combat, exploration, industry; crew-level fidelity as the depth yardstick) in the spec's §1/§2 so "build-for-but-not-with" is judged against the real ambition rather than rung-5. This is a one-paragraph "north star / fidelity ambition" note, not a scope change.

## C. Deltas to apply once the implementation lands

- **Task 13:** replace the synthetic `GOLDEN_ZERO_STATE_HASH` (encodes cursor=0 while hashing 1 body+1 craft — unreachable, since `insert` advances cursor to 1) with a golden computed from `state_hash()` over a fixed committed `RunConfig` at true post-reset cursors (1/1). Keep a **separate** pure-encoding unit test pinning the word-stream→`finish()` mapping. Recapture golden ONLY alongside a `HASH_FORMAT_VERSION` bump, never to chase code.
- **integrator.rs:** verify `substep_count`'s input is a conservative **within-tick maximum** accel (max of entry, next-tick body geometry, full-throttle thrust at min mass), not the tick-entry value — else a low-entry/high-mid-tick periapsis pass gets too few substeps. Add the close-periapsis case to §8.
- **integrator.rs:** add a benchmark at `max_substeps` for expected `num_envs×num_craft×num_bodies`; fill §5.3's asserted-but-empty worst-case substep budget with the formula + measured baseline; pin a `max_substeps` default.
- **ephemeris.rs:** confirm cubic-Hermite (velocity already stored) is the default sub-tick interpolant, so its error order ≥ the integrator's global order. **If interpolation order can vary at runtime, fold it into `config_hash`** (currently NOT in `RunConfig`/`config_hash` — a real run-identity gap: two interp orders share a `config_hash` but produce different trajectories, colliding on replay identity).
- **jumpgate-py FFI:** verify `reset`/`step` check buffer length == `N_envs*N_craft*obs_dim` (and `action_dim`), require C-contiguous dtypes, return a clean `PyErr` ValueError (never a Rust panic across FFI) on mismatch; state the no-alias precondition between the action input and the four output buffers. Add Python tests for wrong-length/non-contiguous buffers. (Note: `PyReadwriteArray1` is a *safe* abstraction — worst case is panic/wrong-data, not UB.)
- **obs.rs:** tighten the frame-relative f32 guard (the spec's self-described highest-confidence invariant). Add a named magnitude const, a positive test at large ego radius (~30 AU) asserting retained precision, and a negative test that the guard fires on a deliberately-absolute coordinate. Decide whether the guard is debug-only (`debug_assert` is compiled out in release/training) or a checked downcast.
- **rng.rs:** verify master→per-stream derivation is a pure function of `(master_seed, stable stream-tag)` with domain separation — NOT index-from-enum, NOT drawn from a parent — so a future stream never perturbs existing seeds.
